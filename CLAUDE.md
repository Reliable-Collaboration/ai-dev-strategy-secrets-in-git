# CLAUDE.md

## Critical Rules

- **NEVER write secrets, credentials, API keys, passwords, tokens, or connection strings into any file outside of `secrets/unencrypted/`.**
- **NEVER commit, stage, or suggest staging files in `secrets/unencrypted/`** or any `.env` / `.env.*` file.
- **NEVER hardcode secret values in source code, configuration files, CI/CD pipelines, Dockerfiles, or documentation.** Use environment variable references instead.
- **NEVER log, print, or echo secret values.** To confirm a variable exists: `echo "API_KEY is set: ${API_KEY:+yes}"`
- **NEVER use `sops edit`.** Always use the decrypt → edit → encrypt workflow via the convenience scripts.
- **Flag suspicious files** (`.env.local`, `*.pem`, `*.key`, `*credentials*`, `*secret*`) found outside `secrets/` for review.

## Project Overview

This repository defines the **Secrets in Git** strategy — a way to store SOPS+age encrypted secrets directly in git repositories. It is distributed as a git submodule into consuming projects.

Full specification: [secrets-in-git.md](./secrets-in-git.md)
Design history (read before making changes): [secrets-in-git-ICD.md](./secrets-in-git-ICD.md)

## Repository Structure

- `secrets-in-git.md` — The strategy specification (follows [Intentional Specification](../intentional-specification/intentional-specification.md))
- `secrets-in-git-ICD.md` — Intentions, Considerations, and Decisions (design reasoning)
- `init.sh` — Scaffolds the secrets structure in a consuming project
- `scripts/encrypt.sh` — Template script: encrypts YAML files in `unencrypted/` to `encrypted/` (single file or all)
- `scripts/decrypt.sh` — Template script: decrypts `.enc.yaml` files in `encrypted/` to `unencrypted/` (single file or all)
- `scripts/dotenv.sh` — Template script: converts a decrypted YAML file to `.env` at the project root

## How It Works (Summary)

- Each developer has their own **age keypair**. Public keys are listed in `.sops.yaml` (committed). Private keys stay on each developer's machine (`~/.config/sops/age/keys.txt`).
- A **break-glass master keypair** provides disaster recovery. Its public key is in `.sops.yaml`; its private key is stored in an external secure location (never in the repo).
- **SOPS** encrypts individual values within YAML files (not whole-file encryption), so git diffs show which keys changed.
- Secrets files must be **flat YAML** (no nesting) for `.env` compatibility.
- The workflow is always: `decrypt.sh [name]` → edit plaintext → `encrypt.sh [name]` → commit encrypted files.
- Use single-file mode (`encrypt.sh dev`) for daily work to avoid re-encrypting unchanged files.

## File Structure in Consuming Projects

```
project-root/
  .sops.yaml                  # Recipient public keys (committed)
  .env                        # Generated by dotenv.sh (.gitignored)
  secrets/
    unencrypted/              # Plaintext secrets (.gitignored)
    encrypted/                # SOPS-encrypted secrets (committed)
    encrypt.sh, decrypt.sh, dotenv.sh
```

## Key Conventions

- Encrypted files use `.enc.yaml` extension; plaintext files use `.yaml`
- One file per environment: `dev.yaml`/`dev.enc.yaml`, `prod.yaml`/`prod.enc.yaml`, etc.
- All environments use the same recipient keys (no per-environment access control)
- All three scripts verify `.gitignore` is protecting both `secrets/unencrypted/` and `.env` before proceeding
- **Read the ICD before modifying the specification.** Do not rewrite existing ICD sections; append new ones.
